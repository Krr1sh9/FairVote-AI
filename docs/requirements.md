# Requirements

This document records the project requirements that are implemented or directly supported by the current repository. It is intended to make the project scope auditable before reading the code.

The project is a research prototype and benchmark for privacy-preserving polling under Local Differential Privacy. It is not production election software.

## Requirement status key

| Status | Meaning |
|---|---|
| Implemented | Code and tests exist in the repository. |
| Implemented with limitations | The feature exists, but the limitation must be stated in the report/viva. |
| Evidence pending | The pipeline exists, but final evidence must be generated by running the configured experiment preset. |
| Out of scope | Explicitly not attempted by this project. |

## Functional requirements

| ID | Requirement | Rationale | Status | Primary implementation |
|---|---|---|---|---|
| FR1 | Provide a respondent-facing poll page that loads a poll configuration and lets a user submit one categorical answer with demographic fields. | The system needs an end-to-end collection prototype, not only offline simulations. | Implemented | `respondent/index.html`, `respondent/static/app.js`, `respondent/server.py` |
| FR2 | Apply k-ary Randomized Response in the browser before the answer is sent to the server. | The privacy mechanism must operate client-side so the server does not need raw answers. | Implemented | `respondent/static/rr.js`; canonical Python reference in `fairvote/privacy/mechanisms/kary_rr.py` |
| FR3 | Store only the perturbed answer, validated demographics, response identifier and reduced-precision timestamp. | Stored respondent data must support aggregate estimation while limiting raw-answer exposure. | Implemented with limitations | `respondent/server.py` |
| FR4 | Provide an aggregate results endpoint that does not expose individual response rows. | The analyst should be able to inspect aggregate outcomes without exporting respondent-level records. | Implemented | `respondent/server.py` `/api/results` |
| FR5 | Protect individual-level response export behind an analyst bearer token. | Individual-level records contain demographics and must not be casually exposed. | Implemented | `respondent/server.py` `/api/responses` |
| FR6 | Parse uploaded CSV and JSONL poll data in the dashboard. | The dashboard must support simulated and respondent-server output workflows. | Implemented | `app/parsing/upload.py` |
| FR7 | Estimate aggregate distributions using raw reported proportions, RR debiasing, RR-aware linear poststratification/MRP, RR-aware Neural MRP and ablation/oracle methods where available. | The research question requires comparisons between simple and more complex estimators. | Implemented | `fairvote/privacy/estimators.py`, `fairvote/inference/mrp/`, `experiments/pipeline/methods/`, `app/services/inference.py` |
| FR8 | Generate synthetic polling scenarios with simple, nonlinear, nonresponse and misreporting behaviour. | Controlled scenarios are needed to test when RR-aware Neural MRP should and should not help. | Implemented | `experiments/pipeline/scenarios.py`, `experiments/pipeline/context.py`, `experiments/pipeline/sampling.py`, `experiments/pipeline/perturbation.py` |
| FR9 | Produce final-evidence experiment outputs suitable for report tables and viva answers. | The final submission needs reproducible evidence, not one-off manual runs. | Evidence pending | `experiments/mrp_vs_baselines.py`, `experiments/pipeline/` |
| FR10 | Export dashboard summaries and evidence bundles for review. | The project should make results portable and auditable. | Implemented | `app/services/exports.py`, `experiments/build_results_bundle.py` |

## Privacy requirements

| ID | Requirement | Rationale | Status | Primary implementation / evidence |
|---|---|---|---|---|
| PR1 | Raw answer fields must be rejected anywhere in the submitted JSON payload, including nested objects and lists. | A malicious client should not be able to smuggle raw answers into storage. | Implemented | `respondent/server.py`, `tests/test_respondent_server.py` |
| PR2 | Forbidden raw-answer keys must include `true_answer`, `true_choice`, `selected_answer`, `selectedOption`, `raw_vote`, and `raw_answer`. | These are plausible names for unprivatized answers. | Implemented | `respondent/server.py` |
| PR3 | Demographics must be validated against `poll_config.json`. | Unknown or free-text demographic fields can increase disclosure risk and break estimation. | Implemented | `respondent/server.py`, `respondent/poll_config.json` |
| PR4 | CORS must be restricted to explicitly configured origins. | Wildcard CORS is not defensible for a privacy-sensitive server. | Implemented | `respondent/server.py`, `FAIRVOTE_ALLOWED_ORIGINS` |
| PR5 | Request body size must be limited. | Reduces accidental/malicious oversized payloads. | Implemented | `respondent/server.py` |
| PR6 | `/api/results` should expose aggregate counts only. | Aggregate inspection is safer than row-level export. | Implemented | `respondent/server.py` |
| PR7 | `/api/responses` must require an analyst token. | Individual-level perturbed records still contain demographics. | Implemented | `respondent/server.py`, `FAIRVOTE_ANALYST_TOKEN` |
| PR8 | Timestamp precision must be configurable and reduced by default. | Full timestamps can increase linkage/re-identification risk. | Implemented | `respondent/server.py`, `FAIRVOTE_TIMESTAMP_PRECISION` |
| PR9 | Documentation must state that Local Differential Privacy protects answer values, not identity, IP logs, demographics or deployment metadata. | Prevents overclaiming. | Implemented | `docs/privacy_boundary.md`, `docs/ethics_and_privacy.md` |
