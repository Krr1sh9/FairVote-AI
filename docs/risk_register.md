# Risk Register

This risk register uses the five-column format expected for the project progress documentation: risk description, impact on project, likelihood, severity/impact rating, and preventative or mitigation actions. Ratings are Low, Medium or High.

| Risk description | Impact on project | Likelihood | Severity / impact rating | Preventative / mitigation actions |
|---|---|---:|---:|---|
| Final-evidence experiment grid takes too long to run on available hardware. | Evidence may be incomplete or based on weaker medium/smoke settings. | Medium | High | Keep separate `smoke_test`, `medium_evidence` and `final_evidence` presets; log runtime by method; allow configurable trials; run final evidence early; reduce only with documented justification. |
| RR-aware Neural MRP does not outperform RR-aware linear poststratification/MRP. | The AI contribution could appear unjustified if framed as guaranteed improvement. | Medium | Medium | Frame the research question conditionally; include simple and nonlinear scenarios; report paired deltas, win rates and confidence intervals; treat mixed/negative results as valid evidence. |
| Synthetic scenarios fail to represent real polling behaviour. | Results may not generalise to real elections or surveys. | High | Medium | State synthetic-data limitations; use scenarios as controlled benchmarks, not real-election forecasts; avoid production claims; include future-work discussion on real validation data. |
| Local Differential Privacy is misunderstood as complete anonymity. | Privacy claims could be overclaimed and challenged in the viva. | Medium | High | Keep `docs/privacy_boundary.md` as canonical; state that LDP protects answer values, not identity, metadata, IP logs or demographics; include limitations in README and final report. |
| Server accidentally stores raw-answer fields from a malicious client. | Privacy boundary would fail and undermine the project. | Low | High | Recursive forbidden-field rejection; server tests for top-level and nested raw fields; JSONL privacy tests; optional browser-level interception test. |
| Demographic values are free-form or inconsistent with the poll configuration. | Estimation can break and privacy risk can increase through unique values. | Low | Medium | Validate demographics against `poll_config.json`; reject unknown fields and invalid values; enforce length and count limits. |
| `/api/responses` exposes individual-level records without protection. | Sensitive perturbed answers plus demographics could be downloaded by unauthorised users. | Low | High | Require `FAIRVOTE_ANALYST_TOKEN` and `Authorization: Bearer ...`; keep `/api/results` aggregate-only; document this boundary. |
| Browser-side Randomized Response and Python RR calculations drift apart. | Client collection and Python inference would use inconsistent privacy mechanisms. | Low | High | Maintain canonical Python RR implementation; keep JS RR only where needed for client-side privacy; test JS and Python probability agreement. |
| Duplicate MRP implementations produce inconsistent results. | A reviewer cannot tell which model is authoritative. | Low | Medium | Use canonical `LinearRRMRPModel`; keep older paths as compatibility wrappers; document canonical path in `docs/mrp_canonical.md`. |
| Missing optional dependencies prevent examiners from running dashboard, respondent app or experiments. | Reproducibility and engineering quality marks suffer. | Low | Medium | Maintain `pyproject.toml` extras; include `requirements-dev.txt`; document install modes in README; CI verifies key imports. |
| Refactoring breaks dashboard behaviour. | Demonstration quality and usability may suffer. | Medium | Medium | Keep `streamlit_app.py` as thin router; unit-test extracted parsing/export/inference helpers; avoid unnecessary UI changes. |
| Experiment output schema changes silently. | Report tables may not match raw evidence; regression comparisons may be invalid. | Low | High | Use regression tests for raw trials, summaries, manifest/config matching, paired comparisons and required metric columns. |
| Randomness prevents reproducible results. | A reviewer cannot reproduce evidence. | Medium | High | Record config seed, sample seed and perturbation seed; write `config.json` and `manifest.json`; use deterministic options where feasible. |
| Neural training is nondeterministic across CPU/GPU or PyTorch versions. | Small numerical differences could affect comparisons. | Medium | Medium | Record seed/device/runtime; use validation loss and early stopping diagnostics; interpret results over repeated trials rather than single runs. |
| Coverage threshold is challenged as too low. | Test evidence may look weak if only a number is considered. | Medium | Medium | Explain coverage gaps in `docs/test_plan.md`; focus on property/statistical/integration tests for critical invariants; keep coverage reports available. |
| Viva examiner asks about why this is not secure election software. | Poor answer could damage credibility. | Medium | High | Explicitly list non-goals; prepare answer from `docs/privacy_boundary.md`; stress research prototype scope and LDP limitations. |
| Hardware/GPU unavailable for neural final evidence. | Neural runs may be slower or infeasible at high trial counts. | Medium | Medium | CPU-compatible training; configurable trials/steps; runtime profiling; use medium evidence if needed with honest limitation; run final evidence early on available resources. |
| Report writing starts too late. | Strong code evidence may not translate into high marks. | Medium | High | Maintain traceability docs, design decisions, experiment protocol and evidence interpretation now; use generated tables and documentation as report scaffolding. |
